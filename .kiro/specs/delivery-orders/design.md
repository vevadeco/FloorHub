# Design Document: Delivery Orders

## Overview

The Delivery Orders feature adds a dedicated dashboard section for managing delivery-type invoices. It mirrors the installation-jobs pattern: invoices with `job_type = 'delivery'` are surfaced in a new `/delivery-orders` page, each gets a unique `DO-XXXX` identifier stored in a `delivery_orders` table, and users can generate a pricing-free PDF and email it to a delivery company via Resend.

Key design decisions:
- Delivery orders are derived from existing invoices — no new invoice creation flow needed.
- The `delivery_orders` table is upserted lazily (on first access) to avoid migration complexity.
- The PDF omits all pricing fields; it shows only product name, sqft, and boxes — suitable for handing to a driver.
- Sequential `DO-XXXX` IDs are generated by querying `MAX(do_number)` from the table, consistent with how invoice numbers are generated elsewhere.

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│  Browser                                                │
│  app/(dashboard)/delivery-orders/page.tsx               │
│    ├── GET /api/delivery-orders          (list)         │
│    ├── PUT /api/delivery-orders/[id]     (upsert job)   │
│    ├── GET /api/delivery-orders/[id]/pdf (download PDF) │
│    └── POST /api/delivery-orders/[id]/send-email        │
└─────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│  API Layer (Next.js Route Handlers)                     │
│    lib/delivery-pdf.ts  — PDF generation                │
│    lib/db.ts            — Neon PostgreSQL               │
│    Resend SDK           — email delivery                │
└─────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│  Database (Neon PostgreSQL)                             │
│    invoices + invoice_items  (existing)                 │
│    delivery_orders           (new)                      │
│    settings                  (existing, Resend config)  │
└─────────────────────────────────────────────────────────┘
```

The feature follows the same layered pattern as installation-jobs: a single list API that joins invoices with delivery_orders, a PUT endpoint that upserts the delivery_orders row, and separate pdf/send-email sub-routes.

## Components and Interfaces

### Pages

**`app/(dashboard)/delivery-orders/page.tsx`**
- Client component, mirrors `installation-jobs/page.tsx` structure
- State: `orders` (list), `selected` (open dialog), `form` (editable fields), `emailInput` (recipient override), `search`, `statusFilter`
- Dialog for editing delivery date, status, notes, and recipient email
- Buttons: Download PDF, Send to Delivery Company

### API Routes

| Route | Method | Purpose |
|---|---|---|
| `/api/delivery-orders` | GET | List all invoices with `job_type='delivery'`, joined with delivery_orders |
| `/api/delivery-orders/[id]` | PUT | Upsert delivery_orders row for invoice `id`; auto-creates DO-XXXX on first insert |
| `/api/delivery-orders/[id]/pdf` | GET | Generate and return Delivery_PDF |
| `/api/delivery-orders/[id]/send-email` | POST | Generate PDF and send via Resend |

### Sidebar

**`lib/delivery-pdf.ts`**

- No pricing columns — table shows: Product, Sq Ft, Boxes

## Data Models

### `delivery_orders` table

```sql
CREATE TABLE IF NOT EXISTS delivery_orders (
  id              TEXT PRIMARY KEY,
  invoice_id      TEXT NOT NULL REFERENCES invoices(id) ON DELETE CASCADE,
  invoice_number  TEXT NOT NULL,
  do_number       INTEGER NOT NULL UNIQUE,   -- sequential, used to build DO-XXXX
  delivery_order_id TEXT NOT NULL UNIQUE,    -- e.g. "DO-0001"
  customer_name   TEXT NOT NULL,
  delivery_date   TEXT DEFAULT '',
  notes           TEXT DEFAULT '',
  status          TEXT NOT NULL DEFAULT 'pending',
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(invoice_id)
)
```

Statuses: `pending`, `scheduled`, `in_transit`, `delivered`, `cancelled`

### TypeScript types (additions to `types/index.ts`)

```ts
export interface DeliveryOrder {
  id: string
  invoice_id: string
  invoice_number: string
  do_number: number
  delivery_order_id: string   // "DO-0001"

  customer_address: string    // from invoice
  delivery_date: string
  notes: string
  status: 'pending' | 'scheduled' | 'in_transit' | 'delivered' | 'cancelled'
  created_at: string
  updated_at: string
}

export interface DeliveryOrderWithItems extends DeliveryOrder {
  items: InvoiceItem[]
  invoice_total: number
}

// Shape returned by GET /api/delivery-orders
export interface DeliveryOrderListItem {
  // invoice fields
  id: string                  // invoice id
  invoice_number: string
  customer_name: string
  customer_address: string
  status: string              // invoice status
  created_at: string
  // joined delivery_orders fields (null if not yet created)

}
```

### DO-XXXX ID generation

```ts
// On INSERT into delivery_orders:
const maxResult = await sql`SELECT COALESCE(MAX(do_number), 0) AS max FROM delivery_orders`
const nextNum = maxResult.rows[0].max + 1
const deliveryOrderId = `DO-${String(nextNum).padStart(4, '0')}`
```

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system — essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: DO-XXXX IDs are unique and sequential

*For any* sequence of delivery order insertions, each `delivery_order_id` must be unique and the `do_number` values must form a strictly increasing sequence with no gaps from the starting point.

**Validates: Requirements 2.2, 2.3**

---

### Property 2: Delivery PDF contains no pricing

*For any* delivery order with associated invoice items, the generated PDF byte stream must not contain any of the strings: "unit_price", "$", "subtotal", "tax", "total", "discount" (case-insensitive).

**Validates: Requirements 4.3**

---

### Property 3: Delivery PDF contains required fields

*For any* delivery order, the rendered PDF must contain the `delivery_order_id`, the customer name, the customer address, and for each item the `product_name` and `boxes_needed` value.

**Validates: Requirements 4.2, 4.4**

---

### Property 4: Upsert idempotence

*For any* delivery order that already exists, calling PUT `/api/delivery-orders/[id]` multiple times with the same payload must result in the same `delivery_order_id` — it must never create a second DO-XXXX for the same invoice.

**Validates: Requirements 2.1, 2.3**

---

### Property 5: Status filter correctness

*For any* status value and list of delivery orders, filtering by that status must return exactly the orders whose `status` field equals that value — no more, no fewer.

**Validates: Requirements 6.4**

---

### Property 6: Email requires Resend API key

*For any* send-email request where `resend_api_key` is empty in settings, the system must return an error response and must not call the Resend API.

**Validates: Requirements 5.4**

---

### Property 7: Email requires recipient address

*For any* send-email request where the recipient email is empty, the system must return a validation error before attempting to send.

**Validates: Requirements 5.5**

---

## Error Handling

| Scenario | Behavior |
|---|---|
| Invoice not found (pdf/send-email) | 404 JSON response |
| Delivery order not found | 404 JSON response |
| `resend_api_key` not configured | 400 with user-friendly message |
| Recipient email empty | 400 validation error |
| Resend API error | 502 with Resend error message surfaced to UI |
| Auth failure | 401 via `AuthError` |
| DB error | 500 with logged error |

All API routes follow the existing `try/catch` pattern with `AuthError` check first, matching `app/api/installation-jobs/[id]/send-email/route.ts`.

## Testing Strategy

### Unit Tests

- `generateDeliveryPDF`: verify the returned buffer is non-empty for a minimal order
- DO-XXXX formatting: `DO-0001`, `DO-0099`, `DO-1000` edge cases
- Status badge color mapping: each status maps to the correct CSS class

### Property-Based Tests

Uses **fast-check** (already a common choice for TypeScript PBT).

Each property test runs a minimum of **100 iterations**.

Tag format: `Feature: delivery-orders, Property {N}: {property_text}`

**Property 1 test** — `// Feature: delivery-orders, Property 1: DO-XXXX IDs are unique and sequential`
Generate N random upsert calls for distinct invoice IDs, assert all resulting `delivery_order_id` values are unique and `do_number` values are 1..N.

**Property 2 test** — `// Feature: delivery-orders, Property 2: Delivery PDF contains no pricing`
Generate random `DeliveryOrderData` with arbitrary items, call `generateDeliveryPDF`, decode buffer to string, assert no pricing keywords present.

**Property 3 test** — `// Feature: delivery-orders, Property 3: Delivery PDF contains required fields`
Generate random orders, render PDF, assert required strings are present in output.

**Property 4 test** — `// Feature: delivery-orders, Property 4: Upsert idempotence`
For a given invoice ID, call the upsert logic twice with different payloads, assert `delivery_order_id` is identical both times.

**Property 5 test** — `// Feature: delivery-orders, Property 5: Status filter correctness`
Generate a random list of orders with random statuses, apply the filter function, assert all results match the filter and none are missing.

**Property 6 test** — `// Feature: delivery-orders, Property 6: Email requires Resend API key`
For any send-email call with empty `resend_api_key`, assert the response status is 400 and Resend is never invoked.

**Property 7 test** — `// Feature: delivery-orders, Property 7: Email requires recipient address`
For any send-email call with empty recipient, assert the response status is 400.
