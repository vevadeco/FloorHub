# Requirements Document

## Introduction

This feature adds three user-facing security and personalization capabilities to the FloorHub Next.js dashboard:

1. **Dark Mode Toggle** — users can switch between light and dark themes, with the preference persisted across sessions.
2. **Auto Logout on Inactivity** — the session is automatically terminated after 30 minutes of user inactivity to reduce unauthorized access risk.
3. **Two-Factor Authentication (2FA) via Authenticator App** — users can enroll a TOTP-based authenticator app (e.g., Google Authenticator, Authy) and are required to provide a one-time code during login when 2FA is enabled.

All three features are surfaced through the existing Settings page (`app/(dashboard)/settings/page.tsx`) and integrate with the existing JWT-based auth system (`app/api/auth/`).

---

## Glossary

- **Dashboard**: The authenticated area of the application rendered under `app/(dashboard)/`.
- **Theme_Toggle**: The UI control that switches the application between light and dark color schemes.
- **Theme_Preference**: The user's chosen color scheme (light or dark), stored in `localStorage` and applied via a CSS class on the `<html>` element.
- **Inactivity_Timer**: A client-side timer that tracks elapsed time since the last user interaction event.
- **Session**: The authenticated state represented by the `floorhub_token` JWT cookie.
- **Auto_Logout**: The process of clearing the Session and redirecting the user to `/login` after the Inactivity_Timer expires.
- **2FA**: Two-Factor Authentication — a second verification step required at login in addition to email and password.
- **TOTP**: Time-based One-Time Password — a 6-digit code generated by an authenticator app per RFC 6238.
- **TOTP_Secret**: A base32-encoded secret key stored per user, used to generate and verify TOTP codes.
- **QR_Code**: A scannable image encoding the `otpauth://` URI that authenticator apps use to register a TOTP account.
- **Backup_Codes**: A set of single-use recovery codes provided to the user when 2FA is enabled, used if the authenticator app is unavailable.
- **Settings_Page**: The page at `app/(dashboard)/settings/page.tsx` where users manage account and security preferences.
- **Login_Page**: The page at `app/(auth)/login/page.tsx` where users authenticate.
- **Auth_API**: The set of API routes under `app/api/auth/`.
- **Authenticator_App**: A third-party TOTP application such as Google Authenticator or Authy.

## Requirements

### Requirement 1: Dark Mode Theme Toggle

**User Story:** As a dashboard user, I want to toggle between light and dark mode, so that I can use the application comfortably in different lighting conditions.

#### Acceptance Criteria

1. THE Theme_Toggle SHALL be accessible from the Settings_Page under an "Appearance" section.
2. WHEN the user activates the Theme_Toggle, THE Dashboard SHALL switch the active color scheme between light and dark mode without a full page reload.
3. WHEN the user sets a Theme_Preference, THE Dashboard SHALL persist the Theme_Preference in `localStorage` under the key `theme`.
4. WHEN the Dashboard loads, THE Dashboard SHALL read the Theme_Preference from `localStorage` and apply it before the first render to prevent a flash of unstyled content.
5. IF no Theme_Preference is stored in `localStorage`, THEN THE Dashboard SHALL apply the color scheme that matches the operating system's `prefers-color-scheme` media query.
6. THE Theme_Toggle SHALL display the current active theme (light or dark) so the user can identify the current state at a glance.
7. WHEN dark mode is active, THE Dashboard SHALL apply the `dark` CSS class to the `<html>` element, consistent with the existing Tailwind CSS dark mode configuration.

---

### Requirement 2: Auto Logout After 30 Minutes of Inactivity

**User Story:** As a security-conscious user, I want the application to log me out automatically after 30 minutes of inactivity, so that my account is protected if I leave my workstation unattended.

#### Acceptance Criteria

1. WHILE a user has an active Session, THE Inactivity_Timer SHALL reset to zero upon any of the following user interaction events: mouse movement, mouse click, keyboard keypress, or touch event.
2. WHEN the Inactivity_Timer reaches 30 minutes without a reset, THE Auto_Logout SHALL clear the Session cookie by calling `POST /api/auth/logout` and redirect the user to `/login`.
3. WHEN the Auto_Logout is triggered, THE Login_Page SHALL display a message indicating the session expired due to inactivity.
4. THE Inactivity_Timer SHALL operate only while the Dashboard is loaded in an active browser tab; the timer SHALL pause when the document visibility state is `hidden`.
5. WHEN the user returns to a tab where the document visibility state transitions from `hidden` to `visible`, THE Inactivity_Timer SHALL check whether 30 minutes have elapsed since the last recorded interaction and trigger Auto_Logout if so.
6. THE Inactivity_Timer logic SHALL be implemented as a single client-side component mounted in the Dashboard layout so that it applies to all dashboard pages without duplication.

---

### Requirement 3: Two-Factor Authentication via Authenticator App

**User Story:** As a security-conscious user, I want to enable two-factor authentication using an authenticator app, so that my account requires a second verification step even if my password is compromised.

#### Acceptance Criteria

##### 3a — Enrollment

1. THE Settings_Page SHALL display a "Two-Factor Authentication" section showing the current 2FA status (enabled or disabled) for the authenticated user.
2. WHEN a user who has 2FA disabled initiates enrollment, THE Auth_API SHALL generate a unique TOTP_Secret for the user and return a QR_Code URI and a plain-text secret for manual entry.
3. WHEN the TOTP_Secret is generated, THE Auth_API SHALL store the TOTP_Secret in the database associated with the user record but SHALL NOT activate 2FA until the user successfully verifies a TOTP code.
4. WHEN the user submits a 6-digit TOTP code during enrollment, THE Auth_API SHALL verify the code against the pending TOTP_Secret using a ±1 time-step tolerance.
5. WHEN the TOTP code is verified successfully during enrollment, THE Auth_API SHALL mark 2FA as enabled for the user and return a set of 8 single-use Backup_Codes.
6. WHEN the TOTP code submitted during enrollment is invalid, THEN THE Auth_API SHALL return a 400 error with a descriptive message and SHALL NOT enable 2FA.
7. WHEN 2FA enrollment is complete, THE Settings_Page SHALL display the Backup_Codes to the user exactly once with a prompt to save them securely.

##### 3b — Login with 2FA

8. WHEN a user with 2FA enabled submits valid email and password credentials, THE Auth_API SHALL return a `{ requires2FA: true }` response with HTTP 200 instead of issuing a Session token.
9. WHEN the Login_Page receives a `requires2FA: true` response, THE Login_Page SHALL display a TOTP code input step without navigating away from the page.
10. WHEN the user submits a 6-digit TOTP code on the Login_Page, THE Auth_API SHALL verify the code against the user's stored TOTP_Secret with a ±1 time-step tolerance.
11. WHEN the TOTP code is valid, THE Auth_API SHALL issue a Session token and set the `floorhub_token` cookie, completing the login flow.
12. WHEN the TOTP code is invalid, THEN THE Auth_API SHALL return a 401 error and THE Login_Page SHALL display an error message without clearing the email/password fields.
13. WHEN a user submits a Backup_Code instead of a TOTP code, THE Auth_API SHALL accept the Backup_Code as valid, invalidate that specific Backup_Code so it cannot be reused, and complete the login flow.

##### 3c — Disabling 2FA

14. WHEN a user with 2FA enabled initiates disabling 2FA from the Settings_Page, THE Settings_Page SHALL require the user to confirm by entering a valid TOTP code or Backup_Code before proceeding.
15. WHEN the confirmation code is valid, THE Auth_API SHALL disable 2FA for the user, delete the stored TOTP_Secret, and invalidate all remaining Backup_Codes.
16. WHEN the confirmation code is invalid, THEN THE Auth_API SHALL return a 400 error and THE Settings_Page SHALL display an error message without disabling 2FA.
